# 第二纪元中段体验验证（逻辑层）

**日期:** 2026-06-09
**阶段:** 第二纪元中段可行性检查
**状态:** 第一轮逻辑验证完成

---

## 1. 本轮目标

在不依赖完整人工长线游玩的前提下，先用代码、配置和单元测试验证第二纪元中段的几个关键问题：

1. 第一批技能树节点是否形成了清晰的成长节奏。
2. 学派限制与分叉限制是否真正构成“路线选择”。
3. 中段核心技能的效果是否足够明确，能支撑下一轮循环提速。
4. 技术债务系统在第二纪元中段是否已经具备实用性。

本轮属于**逻辑层验证**，不是完整的真实游玩结论。

---

## 2. 本轮验证范围

### 2.1 关键路径成本

基于 `game/paradigms.configs.ts`，当前第二纪元早中段路径成本非常规整：

- 首次进入第二纪元后的入门路线：
  - `system_kernel + 任一学派入门 = 3 SP`
- 第二轮核心节点：
  - `system_kernel + 学派入门 + 第一核心技能 = 7 SP`
  - 例如：
    - `pointer_arithmetic`
    - `design_patterns`
    - `dynamic_typing`
- 第三轮分叉前节点：
  - `system_kernel + 学派入门 + 第一核心技能 + 第二层节点 = 13 SP`
  - 例如：
    - `memory_management`
    - `bit_manipulation`

这个 `3 → 7 → 13` 的节奏是成立的，说明第二纪元前几轮并不是随机堆点，而是存在稳定的路径预期。

### 2.2 中段关键约束

代码中已经明确存在两种中段约束：

- **双学派上限**
  - 三个主学派（效率、抽象、敏捷）最多选择两个。
- **分叉互斥**
  - `design_patterns` 之后的 `polymorphism / dependency_injection`
  - `dynamic_typing` 之后的 `jit_compilation / refactoring_tools`

这说明第二纪元中段的策略结构不是“全都要”，而是明确要求路线取舍。

### 2.3 中段关键技能效果是否够清晰

本轮重点检查了几类“应该直接影响体感”的技能：

- `pointer_arithmetic`
  - 对 1、2 阶生成器提供非常明显的早期再启动加速。
- `dynamic_typing`
  - 购买高阶生成器时会补低一阶，能明显加快生产链回填。
- `code_generation`
  - 同时提升手动超频收益和 Code Rush 充能速度，属于主动玩法强化。

这些节点都不是“看不见的隐性收益”，而是玩家能较快体感到差异的节点，因此第二纪元前中段在“每轮都应该更快”这一点上，基础设计是成立的。

## 3. 技术债务系统的关键发现

这是本轮最重要的结论。

### 3.1 典型债务成本

当前技术债务成本公式为：

`CP Cost = 10 ^ (FrozenSP * 1.5 + 120)`

把几条典型中段路径代入后：

- 只重构 `pointer_arithmetic`：
  - 冻结 `4 SP`
  - 成本 `1e126 CP`
- 重构 `efficiency_starter -> pointer_arithmetic` 这条基础链：
  - 冻结 `6 SP`
  - 成本 `1e129 CP`
- 重构一个第二层分支组合（如 `pointer_arithmetic + memory_management`）：
  - 冻结 `10 SP`
  - 成本 `1e135 CP`
- 重构跨学派联动组合（如 `compiler_optimization` 相关前置）：
  - 冻结 `24 SP`
  - 成本 `1e156 CP`

### 3.2 结论

当前重复技术奇点的基础门槛仍是 `1e120 CP`。

这意味着：

- 技术债务在概念上已经成立。
- 但在**第二纪元早中段**，它极可能还不是一个常规可用的“后悔药”。
- 更准确地说，它更像一个偏后期的重规划工具，而不是玩家第二、第三次奇点后就能自然使用的系统。

### 3.3 风险判断

这会带来一个潜在体验问题：

- 文案和系统定位会让玩家以为“学错了可以改”
- 但真实数值门槛可能让他们在较长时间内根本用不起

如果不额外说明，玩家可能把它理解为：

- 不是“昂贵但可用”
- 而是“理论上存在，实际上很难碰”

这点值得在后续中段实玩里重点验证。

## 4. 本轮新增验证用例

本轮已补充以下单元测试，作为第二纪元中段的回归基线：

1. 双学派上限生效
2. 分叉互斥规则生效
3. `pointer_arithmetic` 对 1、2 阶生成器形成明确加速
4. `dynamic_typing` 能回填低一阶生成器
5. `code_generation` 同时提升手动收益和 Code Rush 充能

这些测试的意义不是锁死最终平衡，而是锁定：

- 路线限制是否还在
- 关键节点是否仍然“有体感”

## 5. 当前判断

### 5.1 成立的部分

- 第二纪元前中段的 `3 → 7 → 13 SP` 成长节奏是清晰的。
- 路线限制与互斥关系可以构成真实选择。
- 第一批核心技能具备可感知收益，能支撑“下一轮更快”的核心承诺。

### 5.2 最大风险

- **技术债务的实用时机偏后。**

如果后续真实游玩证明玩家在第二纪元中段几乎从不使用技术债务，那么系统虽然“功能正确”，但会在体验定位上失真。

---

## 6. 下一步建议

### 6.1 最优先

开始一次**第二纪元中段真实游玩验证**，重点观察：

1. 第二、第三次奇点后，玩家是否自然能走到 `7 SP` 与 `13 SP` 节点。
2. 玩家是否会真实地在中段考虑使用技术债务。
3. 如果不会，原因是：
   - 成本过高
   - 说明不清楚
   - 还是当前路线选择其实已经足够稳定，不需要洗点

### 6.2 如果下一轮验证证明技术债务过晚

可考虑的调整方向有三种：

1. 下调债务公式的底数或指数
2. 为第二纪元中段增加债务减免来源
3. 明确将技术债务重新定位为“偏后期重规划系统”，并在文案上避免玩家过早期待

---

## 7. 结论

第二纪元中段在“成长路径”和“节点体感”上，基础结构是成立的。

当前最值得警惕的不是技能树本身，而是**技术债务的可用时机**。

因此，下一轮工作的重点不应是继续扩展第三纪元，而应是：

- 做一次真实的第二纪元中段游玩验证
- 重点确认技术债务到底是“贵但好用”，还是“存在但太晚”
