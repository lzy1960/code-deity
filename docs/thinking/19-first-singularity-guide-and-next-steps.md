# 首次奇点攻略与后续开发建议

**日期:** 2026-06-09
**主题:** 第一次进入第二纪元的实战策略沉淀
**状态:** 已记录

---

## 1. 目的

本文档用于沉淀两类信息：

1. 给当前玩家使用的“首次技术奇点”实战攻略。
2. 给后续开发使用的、基于当前版本状态整理的优先级建议。

需要特别说明的是：

- 设计总纲中仍能看到第一纪元目标为 `1e308 CP` 的旧表述。
- 但**当前实际实现**的第一次技术奇点门槛，已经调整为：
  - `CP >= 1e120`
  - `有效奇点筹备度 >= 100`

因此，本攻略以当前代码实现为准，而不是以旧设计稿中的长期目标为准。

---

## 2. 当前实现下的第一次奇点条件

### 2.1 解锁条件

第一次进入第二纪元前，玩家必须同时满足：

- 算力达到 `1e120`
- 有效奇点筹备度达到 `100`

其中“有效奇点筹备度”采用保底模型：

- 它会取 `breakthroughReadiness` 与 `earnedBreakthroughReadiness` 中的较大值。
- 即使旧存档中的筹备度字段没有及时更新，只要玩家已经完成足够的发布或挑战，系统仍会正确判定可进入奇点。

### 2.2 首次奇点收益

- 技术奇点会进行一次软重置，清空当前 CP 和生成器数量。
- 玩家会获得新的资源 `SP (奇点算力)`。
- 当前版本下，首次奇点**至少获得 3 SP**。

这意味着玩家首次进入第二纪元后，至少可以立刻购买：

- `system_kernel`
- 一个学派入门节点

这也是第二纪元开局体验成立的关键基础。

---

## 3. 首次奇点前的推荐推进路线

### 3.1 总体目标

第一次奇点前，不要只盯着 `CP`。

更准确的目标应该是同时推进两条线：

1. 主线资源线：把 `CP` 推到 `1e120`
2. 里程碑线：把奇点筹备度推到 `100`

如果只刷 CP 而忽略筹备度，最后会出现“资源够了但不能点奇点”的落差感。

### 3.2 前中期：尽快进入发布循环

推荐优先完成以下节奏：

1. 稳定解锁 `Refactor`
2. 尽快推进到第一次 `Compile & Release`
3. 继续推进到 `Version 2`

这样做的原因：

- 只要有 `Version`，奇点筹备度面板就会稳定进入主流程。
- 自动化在第一次发布后解锁，能显著降低重复操作负担。
- 挑战在 `Version >= 2` 后解锁，而挑战是奇点筹备度最强的一批来源之一。

### 3.3 中后期：用“发布 + 挑战”堆筹备度

当前版本中，奇点筹备度主要来自三类来源：

- 一次 `Compile & Release`：`+20`
- 完成一次挑战：`+25`
- 一次高质量 `Refactor`（单次获得至少 `10 RP`）：`+8`

因此，第一次奇点前最稳的路线不是无脑刷重构，而是：

- 以发布为主干
- 以挑战为大块推进
- 用高质量重构补足缺口

这是当前最接近“短回合也有反馈，长回合也有目标”的推进方式。

### 3.4 遇到 AI 核心墙时，不要死堆

当 `AI Core` 拥有量超过 `25` 后，游戏会开始施加“架构过载”惩罚。

这会导致一个常见误区：

- 玩家感觉自己已经进入终局冲刺阶段，于是持续堆最高阶生成器。
- 但实际上总 CPS 会越来越不划算，推进节奏反而变钝。

推荐处理方式：

- 如果继续堆 `AI Core` 的收益明显变差，就应该考虑重构或发布。
- 不要把“继续堆 AI Core”当成唯一解。

这条建议本质上是在顺应系统设计，而不是对抗系统设计。

### 3.5 主动玩法的最佳使用时机

当前版本中，手动超频仍然很有价值，尤其是在以下时机：

- 快解锁下一阶生成器前
- 快够一次关键重构前
- 快够一次编译发布前
- 奇点前最后一段资源冲刺时

原因：

- 手动超频收益按当前 `CPS` 的比例计算。
- `Code Rush` 在短时间爆发期内，能把手动收益进一步放大。

这意味着它不是“前期点击补点数”的设计，而是“中后期主动冲刺工具”。

## 4. 首次进入第二纪元后的推荐开局

### 4.1 第一次 3 SP 的推荐购买

默认推荐：

- `system_kernel`（1 SP）
- 一个学派入门节点（2 SP）

这样可以保证首次奇点后，玩家立刻获得：

- 一个稳定的全局乘区
- 一个明确的发展方向

### 4.2 默认最推荐的首个方向

如果目标是“尽快让下一轮重新跑起来”，推荐优先考虑**效率学派**：

- `system_kernel`
- `efficiency_starter`
- 下一轮尽快攒到 `pointer_arithmetic`

原因：

- `pointer_arithmetic` 会显著增强前两阶生成器，属于最容易体感到“重开变快了”的技能之一。
- 对第一次进入第二纪元的玩家来说，这种直接、清晰的正反馈最重要。

### 4.3 其他方向的定位

- **抽象学派**更像稳定派，适合想要更平滑联动和后续自动化增强的玩家。
- **敏捷学派**更偏重构效率和主动操作，对理解系统较深的玩家更有吸引力。

第一次奇点后，不建议让玩家面对“完全开放但没有指引”的技能树。
当前版本更适合鼓励玩家先理解“`system_kernel + 一个学派起点`”这套基础心智模型。

## 5. 当前阶段最值得优先推进的开发建议

基于现有文档、代码状态和已完成的第二纪元入口调优，下一阶段建议如下。

### 5.1 优先建议一：先做第二纪元中段实玩验证，不要急着进入第三纪元

这是当前最重要的建议。

理由：

- 第二纪元入口已经成立，但中段成长曲线还没有充分验证。
- 当前文档也已经明确建议，先验证第二纪元中段，再进入第三纪元“神祇 / 系统补丁”设计。

建议验证的问题：

1. 第二、第三次奇点是否能稳定拿到足以继续点技能树的 SP。
2. 第一批核心技能是否真的让回合节奏明显变快。
3. 学派上限与分叉限制，玩家感受到的是“策略取舍”还是“被硬卡住”。
4. 技术债务系统在真实游玩中是否足够清晰、足够值得使用。

### 5.2 优先建议二：统一文档与实现，修掉“旧目标残留”

当前仓库里同时存在两套表述：

- 旧设计：第一次奇点目标为 `1e308 CP`
- 现实现：第一次奇点目标为 `1e120 CP + 筹备度 100`

这会在三个层面制造问题：

1. 玩家理解混乱
2. 设计讨论失焦
3. 后续数值调整时容易误用旧假设

建议至少统一以下文档：

- `AGENTS.md`
- `README.md`
- 与奇点/第二纪元目标有关的 `docs/thinking/*` 文档

如果后续仍保留“长期目标 1e308”的大版本设想，也建议明确标注为“远期设计，不代表当前实现”。

### 5.3 优先建议三：补一层游戏内指引，而不只是文档解释

当前系统其实已经具备足够多的策略深度，但玩家首次接触时仍容易出现以下疑问：

- 为什么 CP 够高了还不能进入奇点？
- 为什么 AI 核心越多，后面反而越慢？
- 第一次 3 SP 到底该怎么点？

这些问题仅靠外部文档解决还不够，建议补充轻量级的游戏内引导：

- 在奇点按钮附近显示“还缺什么条件”
- 在第一次获得 3 SP 后给出推荐购买提示
- 在 AI 核心进入过载区间时给出更明确的策略提示

这类工作投入不大，但对第一次进入第二纪元的体验帮助很直接。

### 5.4 优先建议四：继续打磨技术债务的可读性与决策反馈

技术债务机制本身设计得很有主题感，但它天然有理解门槛。

当前风险不在“公式是否存在”，而在于：

- 玩家能不能一眼看懂会失去什么
- 玩家能不能判断现在是否值得偿还
- 玩家会不会因为害怕操作失误而完全不碰这个系统

建议优化方向：

- 在 UI 中更明确展示“将被冻结的技能”“将返还的 SP”“当前需要支付的 CP”
- 增加一个简单的收益预估文案，例如“偿还后你可以立刻改走另一条分支”
- 如果条件允许，补一个首次使用时的引导说明

### 5.5 优先建议五：建立第二纪元数值验证用的半自动回归样例

第二纪元的策略空间已经显著大于第一纪元，仅靠主观试玩会越来越难判断问题来自哪里。

建议下一阶段补一套轻量验证样例，例如：

- 首次奇点后 3 SP 的默认路线样例
- 第二次奇点前的效率学派路线样例
- 抽象/敏捷路线的对照样例
- 技术债务重构一次后的回合恢复速度样例

不一定要做成完整模拟器，但至少可以把关键阶段的存档、目标和预期区间记录下来，作为后续改平衡时的回归基线。

---

## 6. 建议的实际开发顺序

如果按当前收益/风险比排序，建议顺序如下：

1. 第二纪元中段实玩验证与记录
2. 文档与实现统一
3. 奇点/技能树的轻量级游戏内引导
4. 技术债务 UI 与可读性增强
5. 第二纪元数值回归样例
6. 第三纪元概念继续扩展

这个顺序的核心思路是：

- 先确认“现在已经做出来的东西”真的成立
- 再去增加新的复杂系统

这样更稳，也更符合本项目“慢节奏、重沉淀”的产品哲学。

---

## 7. 结论

当前版本下，第一次技术奇点已经具备了明确、可执行的推进路径：

- 玩家需要同时经营 `CP` 和奇点筹备度。
- 发布与挑战是第一次奇点前最重要的里程碑推进器。
- 首次 3 SP 的开局选择已经足以建立第二纪元的基础策略感。

而从开发角度看，当前最有价值的工作不是立刻扩展第三纪元，而是：

- 继续验证第二纪元中段是否真正好玩
- 清理文档与实现的认知偏差
- 把关键策略提示从“仓库文档”前移到“游戏内反馈”

只要这三件事做好，后续第三纪元的设计基础会稳很多。
