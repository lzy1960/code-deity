# 验证记录：奇点引导 UI 第一轮落地

**日期:** 2026-06-09
**阶段:** 第一纪元到第二纪元入口引导增强
**状态:** 第一轮落地完成

---

## 1. 本轮目标

在不扩展第二纪元新系统的前提下，补齐玩家从“接近奇点”到“第一次进入编程范式”的关键引导信息，降低以下理解成本：

1. 为什么我现在还不能点技术奇点？
2. 第一次奇点到底缺的是 CP 还是筹备度？
3. 第一次拿到 3 SP 后应该先点什么？
4. AI 核心过载后，我现在应该继续堆，还是该重构 / 发布？

---

## 2. 已落地内容

### 2.1 奇点筹备度面板增加“双条件”显示

在 `BreakthroughReadinessPanel.vue` 中补充：

- `CP` 条件显示
- 首次奇点所需的筹备度条件显示
- 每项条件是否达成
- 当前阶段提示文案

并且提示逻辑区分了：

- **首次奇点**：需要 `1e120 CP + 筹备度 100`
- **后续奇点**：只显示 `CP` 条件，避免误导玩家继续追筹备度

### 2.2 技术奇点确认弹窗增加条件摘要

在 `SingularityConfirmationModal.vue` 中补充：

- 当前 `CP` 条件摘要
- 首次奇点时的筹备度条件摘要
- 首次奇点后的推荐开局

当前默认推荐路线为：

- `System Kernel`
- `Efficiency Starter`
- 下一轮优先推进 `Pointer Arithmetic`

### 2.3 编程范式页增加首次 3 SP 推荐卡

在 `SingularitySection.vue` 中增加首次进入第二纪元时的推荐说明，仅在以下条件下显示：

- 已进入第二纪元
- 当前为第一次奇点
- 还没有购买任何范式

这样可以避免它在后续循环中持续打扰玩家。

### 2.4 架构过载提示改为“带建议”的反馈

在 `StatsSection.vue` 与首页过载点击反馈中补充更明确的策略建议：

- 未进入发布循环时，优先建议考虑 `Refactor`
- 已进入发布循环时，优先建议在 `Refactor / Compile & Release` 间选择
- 已接近奇点时，将过载解释为最后冲刺成本，而不是长期停留状态

这让“架构过载”从纯状态提示，变成了一个更有行动性的指引。

## 4. 当前结论

本轮引导增强已覆盖玩家在奇点入口最容易卡住的三类问题：

1. **门槛不透明**
2. **首次 3 SP 不知道怎么点**
3. **AI Core 过载时不知道现在该干嘛**

从代码层面看，这轮改动属于“低风险高收益”的可见性优化，没有引入额外系统耦合，也没有改动核心数值逻辑。

---

## 5. 残余风险

当前仍有一个明显空白：

- 本轮主要完成了**编译、类型、测试层面的验收**
- 尚未完成一次真实浏览器中的可视化冒烟检查

因此仍建议在后续合适时机补一轮人工检查，重点看：

1. 移动端下奇点条件卡片是否挤压
2. 首次 3 SP 推荐卡是否与技能树顶部布局冲突
3. 奇点确认弹窗在中英文下是否有溢出
4. 架构过载建议在 stats 页中是否足够醒目

当前最合理的后续顺序：

1. 进行一次轻量人工 UI 冒烟检查
2. 开始第二纪元中段实玩验证
3. 根据第二纪元中段体验，再决定是否继续推进“技术债务”可读性增强或第三纪元设计
