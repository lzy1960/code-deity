## 2. 本阶段关键修正

### 2.1 奇点筹备度改为有效值模型

原先筹备度依赖 `breakthroughReadiness` 存档字段直接显示。如果编译发布、挑战完成或旧存档迁移路径没有及时写入该字段，UI 可能显示为 `0`。

当前已调整为“有效筹备度”模型：

- `earnedBreakthroughReadiness`: 根据已发布版本数和已完成挑战数推导保底筹备度。
- `effectiveBreakthroughReadiness`: 取实际筹备度和保底筹备度中的较大值。
- UI 面板与技术奇点可用性判断均读取有效筹备度。

这样即使旧存档字段为 `0`，只要玩家已经完成编译发布或挑战，界面和奇点判断仍能正确反映进度。

### 2.2 AI 核心压力变为可感知增长

AI 核心压力原增长速率过低，且 UI 只显示整数，导致玩家几乎无法感知筹备度变化。

当前已调整：

- 提高 AI 核心压力增长速率。
- UI 支持一位小数显示。
- 继续保留编译发布、挑战完成、高价值重构等离散进度来源。

### 2.3 编译发布反馈链路加固

编译发布现在会通过页面层显式 handler 调用发布逻辑，并在发布后立即同步筹备度，降低运行时绑定或热更新场景下的排查成本。
